elasticSearch学习（三）

发表于 2020-07-08 | 分类于 elasticSearch | |
字数统计字: 1.1k | 阅读时长分钟 ≈ 4

文档

文档元数据

  • _index：表明文档存放在哪里
  • _type: 文档表示的对象类别
  • _id文档唯一标识

索引文档

  • 通过提供id

  • 不添加id的时候，使用自增id

取回文档

  • 使用get,带上元数据参数，get请求会返回整个文档
  • 获取部分文档，可以通过_source指定返回的字段
  • 使用_mget批量获取文档

检查文档是否存在

  • 使用head代替get,只返回一个状态码

更新文档

  • 文档一旦创建是不能更改的，如果需要更新文档，只能重新索引，旧的文档也不会立刻删除，只是无法访问到
  • 部分更新，同样是更新然后重索引，不过是在分片内部进行，减少了交互操作

创建新文档

  • 通过元数据唯一表示一个文档，在创建新文档的时候，通过元数据来区别，判断是否已经存在相同的文档，也可以使用自增id

删除文档

  • 使用delete，不会立即删除文档，在后台清理已标记为删除的文档

乐观并发控制

  • 通过版本号来控制，也可以使用外部版本号，比如数据库的版本号，在更新文档的时候指定

分片

路由主分片

  • 通过hash(routing)%numbers_of_primary，来确定主分片，这也是为什么创建索引后主分片数量不可变的原因，如果主分片变化了，就找不到文档位置了

主分片和副分片交互

  • 因为相同主分片的副本不会存储在同一节点，可以发送请求到集群中任一节点，每个节点都能处理请求，每个节点都知道集群中任一文档的位置

集群中新建，删除，索引文档

  • 客户端向 Node 1 发送新建、索引或者删除请求。
  • 节点使用文档的 _id 确定文档属于分片 0 。请求会被转发到 Node 3，因为分片 0 的主分片目前被分配在 Node 3 上。
  • Node 3 在主分片上面执行请求。如果成功了，它将请求并行转发到 Node 1 和 Node 2 的副本分片上。一旦所有的副本分片都报告成功, Node 3 将向协调节点报告成功，协调节点向客户端报告成功

取回一个文档

  • 客户端向 Node 1 发送获取请求。
  • 节点使用文档的 _id 来确定文档属于分片 0 。分片 0 的副本分片存在于所有的三个节点上。 在这种情况下，它将请求转发到 Node 2 。
  • Node 2 将文档返回给 Node 1 ，然后将文档返回给客户端。

局部更新文档

  • 客户端向 Node 1 发送更新请求。
  • 它将请求转发到主分片所在的 Node 3 。
  • Node 3 从主分片检索文档，修改 _source 字段中的 JSON ，并且尝试重新索引主分片的文档。 如果文档已经被另一个进程修改，它会重试步骤 3 ，超过 retry_on_conflict 次后放弃。
  • 如果 Node 3 成功地更新文档，它将新版本的文档并行转发到 Node 1 和 Node 2 上的副本分片，重新建立索引。 一旦所有副本分片都返回成功， Node 3 向协调节点也返回成功，协调节点向客户端返回成功

多文档

  • 使用_mget
    • 客户端向 Node 1 发送 mget 请求。
    • Node 1 为每个分片构建多文档获取请求，然后并行转发这些请求到托管在每个所需的主分片或者副本分片的节点上。一旦收到所有答复， Node 1 构建响应并将其返回给客户端
  • 使用bulk
    • 客户端向 Node 1 发送 bulk 请求。
    • Node 1 为每个节点创建一个批量请求，并将这些请求并行转发到每个包含主分片的节点主机。
    • 主分片一个接一个按顺序执行每个操作。当每个操作成功时，主分片并行转发新文档（或删除）到副本分片，然后执行下一个操作。 一旦所有的副本分片报告所有操作成功，该节点将向协调节点报告成功，协调节点将这些响应收集整理并返回给客户端
