Redis数据结构详解

发表于 2020-07-09 | 分类于 redis | |
字数统计字: 2.8k | 阅读时长分钟 ≈ 10

Redis中数据结构

redis中常用的数据结构由string list hash set sorted set，还有比如sub/pub等

各种数据结构使用场景

string
  • 计数，由于Redis单线程的特点，我们不用考虑并发造成计数不准的问题，通过 incrby 命令，我们可以正确的得到我们想要的结果
  • 限制次数，比如登录次数校验，错误超过三次5分钟内就不让登录了，每次登录设置key自增一次，并设置该key的过期时间为5分钟后，每次登录检查一下该key的值来进行限制登录。
hash
  • 查询时间复杂度是O(1)，常用作缓存
list
  • 栈，通过命令 lpush+lpop
  • 队列，命令 lpush+rpop
  • 有限集合，命令 lpush+ltrim
  • 消息队列，命令 lpush+rpop
set
  • 利用set的特性，可以方便的获取多个用户的共同好友等
sorted set
  • 利用排序的特点，可以做排行榜等

Redis底层数据结构

可以通过object encoding key查看某个key的底层数据结构，比如string的底层是embstr和int

string的底层实现

  • 底层是c语言实现的，但是不同于c语言的字符串，。被称为简单动态字符串，它包含额外两个变量len和free

  • 为什么要实现自己的字符串数据结构呢

    • 可以常数时间获取字符串长度
    • 因为存在free,一旦空间不满足，就会扩展，不会造成缓冲区溢出
    • 通过空间预分配和惰性空间释放减少了修改字符串的内存分配次数
    • 二进制安全的，因为c语言是通过空字符作为字符串结束标志，而sds是以len属性来判断，即使二进制文件中包含空字符也能正确截取
    • 兼容c字符串函数

链表的底层实现

  • c语言没有链表数据结构，redis是自己实现的

  • 可以看到，是双端链表，且无环

  • 有一个长度计数器

hash底层实现

  • 仍然是自己实现的数据结构

  • 扩容和缩容

    • 如果执行扩展操作，会基于原哈希表创建一个大小等于 ht[0].used*2n 的哈希表（也就是每次扩展都是根据原哈希表已使用的空间扩大一倍创建另一个哈希表）。相反如果执行的是收缩操作，每次收缩是根据已使用空间缩小一倍创建一个新的哈希表。
    • 重新利用上面的哈希算法，计算索引值，然后将键值对放到新的哈希表位置上。
    • 所有键值对都迁徙完毕后，释放原哈希表的内存空间。
    • 触发扩容的条件
      • 服务器目前没有执行 BGSAVE 命令或者 BGREWRITEAOF 命令，并且负载因子大于等于1
      • 服务器目前正在执行 BGSAVE 命令或者 BGREWRITEAOF 命令，并且负载因子大于等于5。

  • 渐近式rehash

    • Redis采用渐进式 rehash,这样在进行渐进式rehash期间，字典的删除查找更新等操作可能会在两个哈希表上进行，第一个哈希表没有找到，就会去第二个哈希表上进行查找。但是进行 增加操作，一定是在新的哈希表上进行的。

跳跃表

  • 搜索：从最高层的链表节点开始，如果比当前节点要大和比当前层的下一个节点要小，那么则往下找，也就是和当前层的下一层的节点的下一个节点进行比较，以此类推，一直找到最底层的最后一个节点，如果找到则返回，反之则返回空。
  • 插入：首先确定插入的层数，有一种方法是假设抛一枚硬币，如果是正面就累加，直到遇见反面为止，最后记录正面的次数作为插入的层数。当确定插入的层数k后，则需要将新元素插入到从底层到k层。
  • 删除：在各个层中找到包含指定值的节点，然后将节点从链表中删除即可，如果删除以后只剩下头尾两个节点，则删除这一层。

整数集合

  • 整数集合的每个元素都是 contents 数组的一个数据项，它们按照从小到大的顺序排列，并且不包含任何重复项。length 属性记录了 contents 数组的大小。
  • 需要注意的是虽然 contents 数组声明为 int8_t 类型，但是实际上contents 数组并不保存任何 int8_t 类型的值，其真正类型有 encoding 来决定

压缩列表

  • 压缩列表并不是对数据利用某种算法进行压缩，而是将数据按照一定规则编码在一块连续的内存区域，目的是节省内存

redis的数据结构底层使用了哪些实现

对象类型

  • 每次创建键值对的时候，至少会创建两个对象：健对象和值对象，redis中对象是通过redisObject的结构保存的

  • 其中type属性保存了对象的类型，即五大数据类型

  • encoding属性决定数据结构，ptr指针指向底层的数据结构

  • lru属性用来保存对象上一次被程序使用的时间，使用当前时间减去这个时间就获得对象空转时间，内存回收策略中也使用到这个属性

  • refcount用于帮助释放内存（引用计数），和共享内存（支持整数值的自负床对象，因为比较是否相等的复杂度是O(1),其他结构比较是否相等复杂度太高）

    • 创建一个新对象，属性 refcount 初始化为1
    • 对象被一个新程序使用，属性 refcount 加 1
    • 对象不再被一个程序使用，属性 refcount 减 1
    • 当对象的引用计数值变为 0 时，对象所占用的内存就会被释放

  • 内存回收策略（当内存使用达到设置的最大值）

    • volatile-lru 利用LRU算法移除设置过过期时间的key (LRU:最近使用 Least Recently Used )
    • allkeys-lru 利用LRU算法移除任何key
    • volatile-random 移除设置过过期时间的随机key
    • allkeys-random 移除随机key
    • volatile-ttl 移除即将过期的key(minor TTL)
    • noeviction noeviction 不移除任何key，只是返回一个写错误 ，默认选项

字符串对象

  • 字符串的encoding可以是int（保存可以用long表示的整数值），raw（保存长度大于44字节的字符串）或者embstr（保存长度小于44字节的字符串）
  • raw需要分配两次内存，而embstr只需要分配一次，但是如果需要增加长度，需要新分配空间，包括redisObject和sdshdr，因此设为只读的
  • int编码保存的值不再是整数或者超出范围的时候，自动转化为iraw，embstr是只读的，如果对其修改，无论是否超过44字节，都转化为raw

list对象

  • 底层使用ziplist和linkedlist
  • 满足以下条件是用ziplist编码，否则使用linkedlist
    • 列表保存元素小于512个
    • 每个元素产犊小于64字节

哈希对象

  • 编码是ziplist或者hashtable
  • ziplist存储的时候将键值对相邻存放
  • 满足以下条件使用ziplist，否则使用hashtable
    • 列表保存元素小于512个
    • 每个元素产犊小于64字节

集合对象

  • 编码方式是intset或者hashtable
  • 使用hashtable的时候，键建是集合中的元素，值都是null
  • 满足下列条件，使用intset，否则使用hashtable
    • 集合对象中所有元素都是整数
    • 集合对象所有元素数量不超过512

有序集合

  • 使用ziplist或者skiplist，使用score排序
  • ziplist 编码的有序集合对象使用压缩列表作为底层实现，每个集合元素使用两个紧挨在一起的压缩列表节点来保存，第一个节点保存元素的成员，第二个节点保存元素的分值。并且压缩列表内的集合元素按分值从小到大的顺序进行排列，小的放置在靠近表头的位置，大的放置在靠近表尾的位置
  • skiplist 编码的有序集合对象使用 zet 结构作为底层实现，一个 zset 结构同时包含一个字典和一个跳跃表，因为要兼顾具体score操作和范围score操作
  • 满足以下条件使用ziplist，否则使用skiplist
    • 保存的元素数量小于128；
    • 保存的所有元素长度都小于64字节。

具体应用

  • 对于string 数据类型，因为string 类型是二进制安全的，可以用来存放图片，视频等内容，另外由于Redis的高性能读写功能，而string类型的value也可以是数字，可以用作计数器（INCR,DECR），比如分布式环境中统计系统的在线人数，秒杀等。
  • 对于 hash 数据类型，value 存放的是键值对，比如可以做单点登录存放用户信息。
  • 对于 list 数据类型，可以实现简单的消息队列，另外可以利用lrange命令，做基于redis的分页功
  • 对于 set 数据类型，由于底层是字典实现的，查找元素特别快，另外set 数据类型不允许重复，利用这两个特性我们可以进行全局去重，比如在用户注册模块，判断用户名是否注册；另外就是利用交集、并集、差集等操作，可以计算共同喜好，全部的喜好，自己独有的喜好等功能。
  • 对于 zset 数据类型，有序的集合，可以做范围查找，排行榜应用，取 TOP N 操作等
