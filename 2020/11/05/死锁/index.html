死锁

死锁问题

  • 一组阻塞的进程持有一种资源等待获取另一个进程所占有的一个资源

死锁模型

  • 可重复使用的资源
    • 在一个时间只能一个进程使用且不能被删除
    • 进程获得资源，后来释放由其他进程重用
    • 处理器，I/O通道，设备和数据结构
    • 如果每个进程拥有一个资源并请求其他资源，死锁可能发生
  • 使用资源
    • 创建和销毁
    • 在I/O缓冲区的中断，信号，消息。信息
    • 如果接收消息阻塞可能发生死锁
    • 可能 少见的组合时间会发生死锁
  • 资源分配图
    • 一组顶点V和边E的集合
      • V有两种类型
        • 一种是包括系统中所欲的进程
        • 一种是包括系统中的所有资源类型
    • 结论
      • 图中不包含循环，则不会发生死锁
      • 如果有循环，则可能死锁，也不能不会发生死锁

死锁特征（必要不充分条件）

  • 互斥：在一个时间内只有一个进程使用资源
  • 持有并等待：进程保持至少一个资源正在等待获取其他进程持有的额外资源
  • 不可抢占：一个资源只能被进程资源释放，进程已经完成了它的任务之后

死锁处理方法

  • 死锁预防
    • 确保系统永远不会进入死锁状态（操作系统性能差）
    • 实现
      • 互斥：共享资源不是必须的，必须占用非共享资源
      • 占有且等待：分配全部的资源，资源利用率低，可能发生饥饿
      • 不可抢占：分配不到资源则释放已经 持有的资源
      • 循环等待：对资源类型进行排序，按顺序申请资源

  • 死锁避免

    • 最简单和最有效的模式是要求每个进程声明它可能需要的的每个类型资源的最大数目
    • 资源的分配状态通过限定提供与分配的资源数量，和进程的最大需求
    • 死锁避免算法动态检查的资源分配状态，以确保永远不会有一个环形等待状态
    • 安全状态：
      • 针对多有的进程，存在安全序列，如何寻找安全序列？
    • 避免死锁就是避免系统进入不安全状态
    • 常用的死锁避免算法
      • 银行家算法
        • 前提条件
          • 存在多个实例
          • 每个进程都必须能最大限度的利用资源
          • 当一个进程请求一个资源，就不得不等待
          • 当一个进程获得所有的资源就必须在一段油先的时间释放他们
        • 数据结构
          • n=进程数量
          • m=资源类型数量
          • 总需求量Max：n*m矩阵，Max[i,j]=k表示进程i最多请求资源j的k个实例
          • 剩余空闲量A1：长度为m的向量，A1[j]=k，表示k个类型为j的资源实例可用
          • 已分配量A2：n*m矩阵，A2[i,j]=k,表示i进程以获得j资源的k个实例
          • 未来需要量N：n*m矩阵，N[i,j]=k，表示进程i至少需要k个j资源的实例才能完成任务

  • 死锁检测

    • 允许系统进入死锁状态
    • 死锁检测算法（后续）
      • work和finish分别是长度为m和n的向量
      • 检测算法的使用
        • 何时、使用什么样的频率检测以来下面几项
          • 死锁多久可能发生
          • 多少进程需要被回滚
        • 如果检测算法多次被调用，有可能是资源图有多个循环，所以页无法分辨是哪些进程造成死锁
      • 终止进程的过程
        • 终止所欲的死锁进程
        • 在一个时间内终止一个今晨鬼知道死锁消除
        • 终止进程的顺序应该是
          • 进程的优先级
          • 进程运行的时间以及多久时间完成
          • 进程占用的资源
          • 进程还需要的资源
          • 多少进程需要被终止
          • 进程是交互的还是批处理
        • 选择最小成本
        • 回滚，返回到一些安全状态
    • 恢复机制
    • 过程
      • 定期调用检测算法来搜索途中是否存在环
      • 算法需要n^2次操作，n是图的顶点数

  • 死锁恢复

    • 运行系统进入死锁状态，然后恢复（开销大）

  • 现代操作系统最常采用的

    • 忽略这个问题，假装系统中从来没有发生死锁，用于大多数操作系统，包括UNIX

IPC

  • 直接通信
    • 进程必须正确的命名对方
      • send：发送信息到进程P
      • receive：从进程Q接受消息
    • 通信链路的属性
      • ​ 自动建立链路
      • 一条链路恰好对应一对通信进程
      • 每对进程之间只有一个连接存在
      • 连接可以是单向的，但通常是双向的
  • 间接通信
    • 定向从消息队列接收消息
      • 每个消息队列都有一个唯一ID
      • 只有他们共享了一个消息队列，进程才能够通信
    • 通信链路的属性
      • 只有进程共享一个共同的消息队列，才能建立链路
      • 连接可以与许多进程相关联
      • 每对进程可以共享多个通信路
      • 链接可以是单向的，也可以是双向的
    • 操作
      • 创建一个新的消息队列
      • 通过消息队列发送和接收消息
      • 销毁消息队列
    • 原语
      • send
      • receive
  • 消息传递可以是阻塞的，也可以是非阻塞的
    • 阻塞被认为是同步的
    • 非阻塞被认为是异步的

进程间通信

  • 信号
    • 软件中断通知事件处理
    • 接收到信号时会发生什么
      • Catch：指定信号处理函数被调用
      • Ignore：依靠操作系统的默认操作
      • Mask：闭塞信号，因此不会传送
    • 缺点：
      • 不能传输要交换的任何数据
  • 管道
    • 子进程从父进程继承文件描述符
  • 消息队列
    • 消息队列按FIFO来管理消息
  • 共享内存
    • 进程
      • 每个进程都有私有地址
      • 在每个地址空间内，明确的设置了共享内存段
    • 优点
      • 快速，方便的共享数据
    • 缺点
      • 必须同步数据访问
    • 实现：
      • 同一块物理地址映射到不同进程的逻辑地址中
