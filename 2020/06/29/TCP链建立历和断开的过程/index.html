TCP三次握手和四次挥手的过程

三次握手

  • TCP报文格式

    TCP报文格式中包括源端口号和目的端口号，32位的序号和32为的确认序号，以及6位标志位(ACK,FIN,SYN URG PSH RSI)

  • 三次握手

    TCP在建立连接的时候需要进行三次握手

    • 第一次握手

      客户端发送SYN = 1 到服务端，且随机生成一个序列号seq = c

    • 第二次握手

      服务端发送SYN = 1 和 ACK = c+1,且随机生成序列号seq = s

    • 第三次握手

      客户端发送SYN = 0 ACK = s+1，seq = c +1

  • 三次握手各个变量的意义

    • 前两次SYN表示客户端和服务端需要进行同步，包括各自发送报文的序号等，第三次的SYN，由于双方已经完成同步，所以置0
    • 前两次的seq都是客户端和服务端各自随机生成的随机序列号，第三次的seq是客户端生成的seq加1，这也是为什么需要三次握手的重要原因
    • 每次的ACK都是接收到的seq加1，表示对接收到的报文确认

  • 为什么需要三次握手

    • 考虑这样的场景：当完成两次握手之后，客户端断开了链接，服务端不知道这个情况，会一直等待客户端发送数据，造成服务端资源浪费，如果大量这样的请求甚至会造成服务端宕机
    • 第三次握手还是确认已经同步的一个重要步骤，当客户端发送ACK的时候，表明两方已经完成同步（即确认了对方的序列号，可以开始发送数据），此时第三次握手是可以携带数据的

四次挥手

  • 过程

    • 第一次挥手：客户端发送FIN = 1，ACK = s + 1为最近接收到服务端发送数据包的序列号加1，seq = c + 1为客户端最近发送数据包序号加1，此时进入fin_waited状态

    • 第二次挥手：服务端发送ACK = c+ 2

    • 第三次挥手： 服务端发送FIN = 1 seq = s+1，此时服务端进入closed_waited

    • 第四次挥手： 客户端发送ACK = s+ 2，进入time_waited，等待2msl时间，断开链接

  • 为什么需要四次挥手

    因为TCP连接是全双工的，所以需要确认各自均没有数据需要发送给对方才能断开连接

  • 为什么需要等待2msl

    因为客户端发送ACK后无法确认服务端时候收到，因此等待2msl用于确认消息送达服务端，另外，可以释放网络中传输的资源，因为当客户端收到FIN以后，由于网络原因，可能还有资源在传输，以免出现端口复用从而不可预估的错误，需要等待一定的时间。
