输入url到响应页面发生了什么

涉及到的协议及作用
  • DNS
    • 根据输入的url解析出相应的ip地址
    • 流程
      • 浏览器会首先查看本地硬盘的 hosts 文件，看看其中有没有和这个域名对应的规则，如果有的话就直接使用 hosts 文件里面的 ip 地址。
      • 如果在本地的 hosts 文件没有能够找到对应的 ip 地址，浏览器会发出一个 DNS请求到本地DNS服务器 。本地DNS服务器一般都是你的网络接入服务器商提供，比如中国电信，中国移动。
      • 查询你输入的网址的DNS请求到达本地DNS服务器之后，本地DNS服务器会首先查询它的缓存记录，如果缓存中有此条记录，就可以直接返回结果，此过程是递归的方式进行查询。如果没有，本地DNS服务器还要向DNS根服务器进行查询。
      • 根DNS服务器没有记录具体的域名和IP地址的对应关系，而是告诉本地DNS服务器，你可以到域服务器上去继续查询，并给出域服务器的地址。这种过程是迭代的过程。
      • 本地DNS服务器继续向域服务器发出请求，在这个例子中，请求的对象是.com域服务器。.com域服务器收到请求之后，也不会直接返回域名和IP地址的对应关系，而是告诉本地DNS服务器，你的域名的解析服务器的地址。
      • 最后，本地DNS服务器向域名的解析服务器发出请求，这时就能收到一个域名和IP地址对应关系，本地DNS服务器不仅要把IP地址返回给用户电脑，还要把这个对应关系保存在缓存中，以备下次别的用户查询时，可以直接返回结果，加快网络访问。

ping命令的原理和过程

ping的使用
  • 简单来说，ping是用来探测本机与网络中另一主机之间是否可达的命令，如果两台主机之间ping不通，则表明这两台主机不能建立起连接。ping是定位网络通不通的一个重要手段
ping的流程
  • 假设有两个主机，主机A（192.168.0.1）和主机B（192.168.0.2），现在我们要监测主机A和主机B之间网络是否可达，那么我们在主机A上输入命令：ping 192.168.0.2；
  • 此时，ping命令会在主机A上构建一个 ICMP的请求数据包（数据包里的内容后面再详述），然后 ICMP协议会将这个数据包以及目标IP（192.168.0.2）等信息一同交给IP层协议；
  • IP层协议得到这些信息后，将源地址（即本机IP）、目标地址（即目标IP：192.168.0.2）、再加上一些其它的控制信息，构建成一个IP数据包；

数据库知识总结

数据库的结构

一条查询语句是怎么执行的
  • 连接器
    • 首先通过连接器连接到数据库上，如果客户端长时间没有操作，连接会断开，使用wait_timeout设置。默认8小时
    • 尽量少使用长连接
  • 查询缓存
    • 连接建立以后，会先到缓存中查看，如果存在，就直接返回，如果不存在，就继续下面的流程
    • 但是对插入更新频繁的数据库，应该少使用缓存，因为每次更新都会使缓存失效，在8.0版本以后完全删除了这个功能
  • 分析器
    • 如果没有命中缓存，就会真正开始执行语句，包括词法分析等
  • 优化器
    • 优化器决定使用哪个索引，或者使用join的时候，各个表的顺序
  • 执行器
    • 先判断是否有执行权限
    • 有权限，就重复执行逻辑，写入结果集中，直到不符合条件
更新语句执行流程

  • 流程和查询差不多，不过此时所有缓存都会被清空，通过执行器找到需要更新的行，执行更新操作

  • redo log(InnoDB特有的)

    • 在更新操作的时候，会先将记录写入到redo log中并更新内存，此时更新完成，在适当的时候，再将记录更新到磁盘中
    • redo log大小是固定的，是可以循环写的，在write pos h和check point之间写，当write pos 追上check point的时候就需要更新记录到磁盘以腾出空间

  • binlog（server层）

    • binlog只能用用于归档不，没有crash-safe能力

  • 两阶段提交

    • redo log 和 binlog 都可以用于表示事务的提交状态，而两阶段提交就是让这两个状态保持逻辑上的一致
事务的隔离性
  • 脏读，不可重复读，幻读
    • 脏读使读到事务还未提交的记录
    • 不同重复读是同一事务不同阶段读到的结果不一致（主要指更新操作）
    • 幻读是两次读取得到不同的结果（主要是新增操作）
  • 隔离级别
    • 读未提交
    • 读已提交
    • 可重复读
    • 串行化
  • 隔离级别是怎么实现的
    • read view视图，在读已提交中，每个sql都会创建视图，而在可重复读中，是在事务开始的时候创建视图，整个事务都访问这个视图，串行化使用加锁实现。
    • 数据库的mvvc，在每次更改数据后都会生成read view并且记录操作的事务id，每个事务只能访问到比自己id小的版本（事务id是递增的）
  • 事务的启动方式
    • 显示的事务启动，使用begin或者start transaction
    • set autocommit = 0会导致长事务
索引
  • InnoDB索引模型
    • B+树，即每个索引都会对应一颗B+树
    • 根据叶子结点的内容，索引有主键索引和非主键索引，主键索引叶子节点存储整行数据，主键索引也被称为聚簇索引，非主键索引的叶子结点存储的是主键的值
    • 基于非主键索引查询会产生回表
  • 降低回表次数
    • 使用联合索引，实现覆盖索引
  • 最左匹配原则
    • 因为索引是有序的，当匹配到范围查询等条件时，后续索引会失效
锁
  • 全局锁，表锁，行锁

TCP三次握手和四次挥手的过程

三次握手

  • TCP报文格式

    TCP报文格式中包括源端口号和目的端口号，32位的序号和32为的确认序号，以及6位标志位(ACK,FIN,SYN URG PSH RSI)

  • 三次握手

    TCP在建立连接的时候需要进行三次握手

    • 第一次握手

      客户端发送SYN = 1 到服务端，且随机生成一个序列号seq = c

    • 第二次握手

      服务端发送SYN = 1 和 ACK = c+1,且随机生成序列号seq = s

    • 第三次握手

      客户端发送SYN = 0 ACK = s+1，seq = c +1

  • 三次握手各个变量的意义

    • 前两次SYN表示客户端和服务端需要进行同步，包括各自发送报文的序号等，第三次的SYN，由于双方已经完成同步，所以置0
    • 前两次的seq都是客户端和服务端各自随机生成的随机序列号，第三次的seq是客户端生成的seq加1，这也是为什么需要三次握手的重要原因
    • 每次的ACK都是接收到的seq加1，表示对接收到的报文确认

  • 为什么需要三次握手

    • 考虑这样的场景：当完成两次握手之后，客户端断开了链接，服务端不知道这个情况，会一直等待客户端发送数据，造成服务端资源浪费，如果大量这样的请求甚至会造成服务端宕机
    • 第三次握手还是确认已经同步的一个重要步骤，当客户端发送ACK的时候，表明两方已经完成同步（即确认了对方的序列号，可以开始发送数据），此时第三次握手是可以携带数据的

四次挥手

  • 过程

    • 第一次挥手：客户端发送FIN = 1，ACK = s + 1为最近接收到服务端发送数据包的序列号加1，seq = c + 1为客户端最近发送数据包序号加1，此时进入fin_waited状态

    • 第二次挥手：服务端发送ACK = c+ 2

    • 第三次挥手： 服务端发送FIN = 1 seq = s+1，此时服务端进入closed_waited

    • 第四次挥手： 客户端发送ACK = s+ 2，进入time_waited，等待2msl时间，断开链接

  • 为什么需要四次挥手

    因为TCP连接是全双工的，所以需要确认各自均没有数据需要发送给对方才能断开连接

  • 为什么需要等待2msl

    因为客户端发送ACK后无法确认服务端时候收到，因此等待2msl用于确认消息送达服务端，另外，可以释放网络中传输的资源，因为当客户端收到FIN以后，由于网络原因，可能还有资源在传输，以免出现端口复用从而不可预估的错误，需要等待一定的时间。

设计模式（二）

行为型

观察者模式
  • 在对象之间定义一个一对多的依赖，当一个对象状态改变的时候，所有依赖的对象都会自动收到通知
模板模式
策略模式

  • 策略模式定义一族算法类，将每个算法分别封装起来，让它们可以互相替换。策略模式可以使算法的变化独立于使用它们的客户端（这里的客户端代指使用算法的代码）。

  • 策略模式用来解耦策略的定义、创建、使用。实际上，一个完整的策略模式就是由这三个部分组成的。

  • 策略类的定义比较简单，包含一个策略接口和一组实现这个接口的策略类。

  • 策略的创建由工厂类来完成，封装策略创建的细节。策略模式包含一组策略可选，客户端代码如何选择使用哪个策略，有两种确定方法：编译时静态确定和运行时动态确定。其中，“运行时动态确定”才是策略模式最典型的应用场景。

  • 移除繁杂的if-else

职责链模式
  • 将请求的发送和接收解耦，让多个接收对象都有机会处理这个请求。将这些接收对象串成一条链，并沿着这条链传递这个请求，直到链上的某个接收对象能够处理它为止。
  • 职责链模式有两种常用的实现。一种是使用链表来存储处理器，另一种是使用数组来存储处理器，后面一种实现方式更加简单
  • servlet filter和spring interceptor
状态模式
  • 状态机又叫有限状态机，它有 3 个部分组成：状态、事件、动作。其中，事件也称为转移条件。事件触发状态的转移及动作的执行
  • 第一种实现方式叫分支逻辑法。利用 if-else 或者 switch-case 分支逻辑，参照状态转移图，将每一个状态转移原模原样地直译成代码。对于简单的状态机来说，这种实现方式最简单、最直接，是首选。
  • 第二种实现方式叫查表法。对于状态很多、状态转移比较复杂的状态机来说，查表法比较合适。通过二维数组来表示状态转移图，能极大地提高代码的可读性和可维护性。
  • 第三种实现方式叫状态模式。对于状态并不多、状态转移也比较简单，但事件触发执行的动作包含的业务逻辑可能比较复杂的状态机来说，我们首选这种实现方式。
迭代器模式
  • 迭代器是用来遍历容器的，所以，一个完整的迭代器模式一般会涉及容器和容器迭代器两部分内容。为了达到基于接口而非实现编程的目的，容器又包含容器接口、容器实现类，迭代器又包含迭代器接口、迭代器实现类
访问者模式
  • 允许一个或者多个操作应用到一组对象上，解耦操作和对象本身
备忘录模式
  • 在不违背封装原则的前提下，捕获一个对象的内部状态，并在该对象之外保存这个状态，以便之后恢复对象为先前的状态
  • 备忘录模式也叫快照模式，具体来说，就是在不违背封装原则的前提下，捕获一个对象的内部状态，并在该对象之外保存这个状态，以便之后恢复对象为先前的状态。这个模式的定义表达了两部分内容：一部分是，存储副本以便后期恢复；另一部分是，要在不违背封装原则的前提下，进行对象的备份和恢复
命令模式
  • 命令模式用到最核心的实现手段，就是将函数封装成对象。我们知道，在大部分编程语言中，函数是没法作为参数传递给其他函数的，也没法赋值给变量。借助命令模式，我们将函数封装成对象，这样就可以实现把函数像对象一样使用。命令模式的主要作用和应用场景，是用来控制命令的执行，比如，异步、延迟、排队执行命令、撤销重做命令、存储命令、给命令记录日志等等，这才是命令模式能发挥独一无二作用的地方。
解释器模式
  • 解释器模式为某个语言定义它的语法（或者叫文法）表示，并定义一个解释器用来处理这个语法。
中介模式
