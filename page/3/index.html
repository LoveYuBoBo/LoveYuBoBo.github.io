SpringMVC

SpringMVC

三层架构

和Structs2的异同点

  • 相同点
    • 底层都是基于原始Servlet的
    • 都是基于MVC的
    • 处理请求的机制都是一个核心控制器
  • 不同点
    • 入口不同：SpringMVC入口是Servlet，Structs2入口是Filter
    • SpringMvVC基于方法，Structs2基于类

常用注解

  • RequestMapping
    • value和path属性均可指定请求路径
    • method方法指定请求方式
    • params 指定请求参数的条件
    • headers指定限制请求消息头的条件
  • RequestParam 请求中指定名称的参数给控制器中的形参赋值
  • RequestBody：用于获取请求体的内容，get方法不适用
  • PathVaribale 用于绑定url中的占位符
  • RequestHeader 获取请求头消息
  • CookieValue 用于获取指定cookie的值
  • ModelArrtibute
  • SessionAttribute 作用在类上

请求参数的绑定

  • controller使用和前端传递的参数一致的参数接收
  • form表单提交的时候name属性的值应该和JavaBean中属性值一致（使用JavaBean中的set方法封装）
  • 引用类型的封装：使用表达式
  • 解决中文乱码的问题，配置过滤器

类型转换

  • 类型转换需要实现Converter接口
  • 配置自定义类型转换器
    • ConversionServiceFactory

动态代理

动态代理

基于接口的动态代理

  • 要求被代理类至少实现一个接口

  • 使用JDK提供的Proxy类

  • 使用Proxy的newProxyInstance方法创建代理对象

  • 创建代理对象的方法有三个参数，CLassLoader用于指定代理对象的字节码，CLass[] 用于让代理对象和被代理对象有相同的方法，InvocationHandler用于提供增强的代码，一般是该接口的实现类

基于子类的动态代理

  • 要求被代理类不能是最终类
  • 由第三方cglib提供的Enhance类
  • 使用Enhance的create方法
  • create方法有两个参数，Class指定被代理对象的字节码，callback用于提供增强的代码。

Spring

Spring

程序间的依赖

  • 类之间的依赖
  • 方法间的依赖
  • 实际开发中应该做到编译期间不依赖，运行期才依赖
  • 解决耦合思路：
    • 使用反射而不是new创建对象
    • 通过读取配置文件获取创建对象的全限定类名

IOC

  • 获取IOC核心容器

  • ApplicationContext三个常用的实现类

    • ClassPathXmlApplicationContext： 加载类路径下的配置文件，必须在类路径下
    • FileSystemApplicationContext：加载磁盘任意位置有访问权限的文件
    • AnnotationConfigApplicationContext：读取注解创建容器

  • 核心容器创建引发的问题：

    • 使用ApplicationContext，会立即创建，只要读取完配置文件就马上创建，适用于单例对象
    • 使用BeanFactory，会延迟加载，什么时候根据id获取对象，什么时候才真正创建，适用于多例对象

spring对bean的管理细节

创建bean的三种方式
  • 使用默认构造函数，使用标签配置id和class且没有配置其他参数的时候就是使用默认构造函数
  • 使用普通工厂中的方法创建对象（使用某个类中的方法创建对象，并存入spring容器）
  • 使用工厂中的静态方法创建对象并存入spring容器
bean对象的作用范围
  • 标签中的的scope属性，用于指定bean作用范围，取值有singleton（单例），prototype（多例），request（web应用的请求范围），session（web应用的会话范围），global-session（集群环境的会话范围）
bean对象的生命周期
  • 单例对象：
    • 容器创建时产生
    • 容器还在，就一直存活
    • 容器销毁，对象消亡
  • 多例对象
    • 使用对象的时候spring为我们创建
    • 对象只要在使用就一直存在
    • 对象长时间不用且没有其他对象引用该对象的时候，垃圾回收
依赖注入

  • 依赖注入：Dependecy Injection

  • IOC的作用：降低程序间的耦合

  • 依赖关系管理都交给spring维护

  • 当前类需要使用其他类的对象的，有spring为我们提供，我们只需要在配置文件中说明依赖关系的维护就成为依赖注入

  • 依赖注入的数据类型，经常变化的数据不适合依赖注入

    • 基本类型和String
    • 在配置文件或注解中配置过的bean
    • 复杂类型/集合类型

  • 注入方式

    • 使用构造函数
      • 使用标签，在bean标签内部
      • 标签中的属性：type，index，name，value（提供基本和String类型），ref（用于指定的其他bean对象）
      • 使用该方法的优劣势：
        • 注入数据是必须的，否则对象无法创建
        • 改变了bean实例化的方式
    • 使用set方法提供
      • 使用property标签
      • 属性：name，value，ref
      • 优势：创建对象的时候没有明确的限制
      • 弊端：如果某个成员必须有值，则set方法可能没执行

  • 使用注解提供

    复杂集合的注入

Spring注解

用于创建对象的注解
  • Component:把当前类对象存入spring容器中，value属性用于指定bean的id，不写的时候，默认当类名且首字母小写
  • Controller：一般用于表现层
  • Service：一般用于业务层
  • Repository：一般用于持久层i
用于注入数据的注解
  • Autowired:自动按照类型注入，只要容器中有唯一一个bean对象类型匹配就能匹配成功
  • Qualifier:按照类型注入的基础上再按照名称注入，给类成员注入时不能单独使用
  • Resource：按照bean的id注入
  • 以上注入只能应用于其他bean类型的数据，不能注入基本数据和String，集合类型的注入只能使用xml
  • Value:用于注入基本类型和String类型，value用于指定数据的值
用于改变作用范围的注解
  • Scope：指定bean的作用范围，value取值有singleton和prototype
和生命周期相关

  • PreDestroy

  • PostContruct

spring的新注解

  • Configration：指定当前类是一个配置类
    • 当配置类作为AnnotationConfigApplicationContext对象创建的参数时，可以不写
  • ComponentScan：指定spring在创建容器时要扫描的包
    • 其属性value和basePackage一致
  • Bean：把当前方法的返回值作为bean对象存入spring的IOC容器中，name用于指定bean的id，不写的时候，默认值是当前方法的名称。
  • 当注解配置方法的时候，如果方法有参数，spring会在容器中查找是否有匹配的参数
  • Import：用于导入其他的配置类，value属性用于指定其他配置类的字节码，使用Import注解的时候，有Import注解的类就是父注解类
  • PropertySource: 用于指定Properties文件的，value属性指定文件名称和路径，关键字classpath表示类路径下
  • Quaiifier匹配参数注入的时候可以单独使用

整合junit

  • 导入spring整合junit的jar包
  • 使用Junit提供的注解@Runwith把原有的main方法替换成spring提供的
  • 告知运行器，spring和ioc的创建使用注解还是配置文件，且说明位置 @ContextConfiguration，其中location指定xml文件的位置，classes指定注解类的位置

AOP

  • xml配置步骤
    • 把通知bean交给spring管理
    • 使用aop:config标签表明开始配置aop
    • aop:aspect表明配置切面 id属性给切面提供一个唯一标识，ref指定通知bean的id
    • 在aop:aspect内部配置对应的通知类型，method属性指定为哪个方法指定通知类型
    • pointcut指定切入点的表达式，指的是对业务层中哪些方法增强
    • aop:pointcut标签可以写在aop:aspect内部，也可以写在aop:asoect外部，在外部的时候必须配置在切面前面。
    • 通知类型：前置通知，后置通知，异常通知，最终通知，环绕通知
      • 动态代理中环绕通知有明确的切入点方法调用
      • spring提供了一个接口ProceedingJoinPoint，该接口有一个方法proceed(),此方法相当于明确调用切入点方法，接口可以作为环绕通知的方法参数
  • 注解配置

JdbcTemplate

  • 保存，更新，删除：update，唯一的区别是sql语句

事务控制

  • 基于xml的事务控制配置
    • 配置事务管理器
    • 配置事务通知
      • 需要导入事务的约束，使用tx:advice配置事务
      • 其中id给事务通知起一个唯一的标识
      • transaction-manager：给事务通知提供事务管理器引用
    • 配置AOP中的通用切入点表达式
    • 建立事务通知和切入点表达式的关系
    • 配置事务的属性，在tx:advice标签内部配置
      • isolation ：指定事务的隔离级别
      • propagation: 用于指定事务的传播行为，默认是REQUIRED，表示一定会有事务，增删改的时候选择
      • no-rollback-for：用于指定一个异常，产生该异常时，事务不回滚
      • read-only：指定事务是否只读，只在查询的时候在设置为true，默认死false
      • rollback-for：用于指定一个异常，产生该异常时，事务回滚
      • timeout：用于指定事务超时的时间，默认永不超时，指定了数值，以秒为单位
  • 基于注解的事务控制
    • 配置事务管理器
    • 开始事务管理：EnableTransactionManagement
    • 开启spring对注解事务的支持，使用tx:annotation-driven
    • 需要事务支持的地方使用Transactional

旅游网项目

技术选型

web层
  • servlet：前端控制器
  • html：视图
  • filter：过滤器
  • BeanUtils：数据封装工具
  • Jackson：json序列化工具
service层
  • Javamail：发送邮件工具
  • Redis：内存数据库
  • Jedis：java的redis客户端
Dao层
  • Mysql：数据库
  • Druid：数据库连接池
  • JdbcTemplate：jdbc的工具
功能
  • 注册
    • 前端
      • 使用js完成表单校验
      • 使用Ajax完成表单提交
        • 前台使用的是html，只能通过ajax获取数据，不能通过servlet的域来获取
      • 注册成功，跳转页面
    • servlet
      • 获取数据
      • 封装User对象
      • 调用Service完成注册
      • 根据service返回，提示信息，将信息封装为json，设置响应头
    • Service
      • 调用dao根据用户名查询用户
        • 存在，则返回false
        • 不存在，调用dao保存用户信息
    • dao
      • 实现查找用户和保存用户信息的逻辑
  • 登录
    • LoginServlet获取用户信息，调用service查询user，判断用户是否存在，判断用户是否激活
