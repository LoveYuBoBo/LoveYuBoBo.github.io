秒杀系统设计

现在很多电商平台都会有秒杀功能，在短时间内，少数参加秒杀的商品会由大量请求到达。很明显，后端db不可能扛得住这么大的压力，这时，缓存，负载均衡，集群等是解决问题的好办法。

需要解决的问题

  • 高性能
  • 一致性，特别是库存一致性
  • 高可用

细节处理

  • nginx，高性能web服务器，能达到几万的并发，但是tomcat只有几百的并发，这个时候可以水平扩展tomcat
  • 秒杀商品的详情等静态资源存放到cdn中，进一步减轻服务器压力
  • 倒数时间，秒杀系统一般都会有一个秒杀倒计时，但是不可能时时刻刻请求服务端来获取倒计时，这是时候，可以采用本地计时的方式，定时获取后端时间并更新
  • 秒杀按钮，秒杀没开始的时候，按钮应该置灰

限流

限流分为i前端限流和后端限流，尽可能使到达后端的流量使有效流量

  • 前端限流
    • 按钮点击频率限制
  • 后端限流
    • 同一ip一个时间段允许一定频谱的访问
    • 加入消息队列，后端根据消费能力取消息来消费

库存预热

  • redis集群缓存数据，秒杀结束在异步写入数据库
  • 并发问题：使用lua脚本，lua脚本类似事务，有一定的原子性
  • 预扣库存的方式，将订单扔到延时队列中，超时失效

常见的排序算法

各种算法比较

具体分析

冒泡排序

  • 比较相邻的元素。如果前一个比后一个大，就交换它们两个

  • 对每一对相邻元素作同样的工作，直到最后一对；

  • 针对所有的元素重复以上的步骤，除了最后一个；

  • 重复直到排序完成。

  • 为了优化算法，可以设立一个布尔标识，每趟排序开始前设为false，如果该趟排序发生了交换就置为true，如果一趟排序结束标识仍为false表示该趟排序没有发生交换，即数组已经有序，可以提前结束排序

  • 最好情况是本来有序，最坏情况是本来逆序

输入url到响应页面发生了什么

涉及到的协议及作用
  • DNS
    • 根据输入的url解析出相应的ip地址
    • 流程
      • 浏览器会首先查看本地硬盘的 hosts 文件，看看其中有没有和这个域名对应的规则，如果有的话就直接使用 hosts 文件里面的 ip 地址。
      • 如果在本地的 hosts 文件没有能够找到对应的 ip 地址，浏览器会发出一个 DNS请求到本地DNS服务器 。本地DNS服务器一般都是你的网络接入服务器商提供，比如中国电信，中国移动。
      • 查询你输入的网址的DNS请求到达本地DNS服务器之后，本地DNS服务器会首先查询它的缓存记录，如果缓存中有此条记录，就可以直接返回结果，此过程是递归的方式进行查询。如果没有，本地DNS服务器还要向DNS根服务器进行查询。
      • 根DNS服务器没有记录具体的域名和IP地址的对应关系，而是告诉本地DNS服务器，你可以到域服务器上去继续查询，并给出域服务器的地址。这种过程是迭代的过程。
      • 本地DNS服务器继续向域服务器发出请求，在这个例子中，请求的对象是.com域服务器。.com域服务器收到请求之后，也不会直接返回域名和IP地址的对应关系，而是告诉本地DNS服务器，你的域名的解析服务器的地址。
      • 最后，本地DNS服务器向域名的解析服务器发出请求，这时就能收到一个域名和IP地址对应关系，本地DNS服务器不仅要把IP地址返回给用户电脑，还要把这个对应关系保存在缓存中，以备下次别的用户查询时，可以直接返回结果，加快网络访问。

ping命令的原理和过程

ping的使用
  • 简单来说，ping是用来探测本机与网络中另一主机之间是否可达的命令，如果两台主机之间ping不通，则表明这两台主机不能建立起连接。ping是定位网络通不通的一个重要手段
ping的流程
  • 假设有两个主机，主机A（192.168.0.1）和主机B（192.168.0.2），现在我们要监测主机A和主机B之间网络是否可达，那么我们在主机A上输入命令：ping 192.168.0.2；
  • 此时，ping命令会在主机A上构建一个 ICMP的请求数据包（数据包里的内容后面再详述），然后 ICMP协议会将这个数据包以及目标IP（192.168.0.2）等信息一同交给IP层协议；
  • IP层协议得到这些信息后，将源地址（即本机IP）、目标地址（即目标IP：192.168.0.2）、再加上一些其它的控制信息，构建成一个IP数据包；

数据库知识总结

数据库的结构

一条查询语句是怎么执行的
  • 连接器
    • 首先通过连接器连接到数据库上，如果客户端长时间没有操作，连接会断开，使用wait_timeout设置。默认8小时
    • 尽量少使用长连接
  • 查询缓存
    • 连接建立以后，会先到缓存中查看，如果存在，就直接返回，如果不存在，就继续下面的流程
    • 但是对插入更新频繁的数据库，应该少使用缓存，因为每次更新都会使缓存失效，在8.0版本以后完全删除了这个功能
  • 分析器
    • 如果没有命中缓存，就会真正开始执行语句，包括词法分析等
  • 优化器
    • 优化器决定使用哪个索引，或者使用join的时候，各个表的顺序
  • 执行器
    • 先判断是否有执行权限
    • 有权限，就重复执行逻辑，写入结果集中，直到不符合条件
更新语句执行流程

  • 流程和查询差不多，不过此时所有缓存都会被清空，通过执行器找到需要更新的行，执行更新操作

  • redo log(InnoDB特有的)

    • 在更新操作的时候，会先将记录写入到redo log中并更新内存，此时更新完成，在适当的时候，再将记录更新到磁盘中
    • redo log大小是固定的，是可以循环写的，在write pos h和check point之间写，当write pos 追上check point的时候就需要更新记录到磁盘以腾出空间

  • binlog（server层）

    • binlog只能用用于归档不，没有crash-safe能力

  • 两阶段提交

    • redo log 和 binlog 都可以用于表示事务的提交状态，而两阶段提交就是让这两个状态保持逻辑上的一致
事务的隔离性
  • 脏读，不可重复读，幻读
    • 脏读使读到事务还未提交的记录
    • 不同重复读是同一事务不同阶段读到的结果不一致（主要指更新操作）
    • 幻读是两次读取得到不同的结果（主要是新增操作）
  • 隔离级别
    • 读未提交
    • 读已提交
    • 可重复读
    • 串行化
  • 隔离级别是怎么实现的
    • read view视图，在读已提交中，每个sql都会创建视图，而在可重复读中，是在事务开始的时候创建视图，整个事务都访问这个视图，串行化使用加锁实现。
    • 数据库的mvvc，在每次更改数据后都会生成read view并且记录操作的事务id，每个事务只能访问到比自己id小的版本（事务id是递增的）
  • 事务的启动方式
    • 显示的事务启动，使用begin或者start transaction
    • set autocommit = 0会导致长事务
索引
  • InnoDB索引模型
    • B+树，即每个索引都会对应一颗B+树
    • 根据叶子结点的内容，索引有主键索引和非主键索引，主键索引叶子节点存储整行数据，主键索引也被称为聚簇索引，非主键索引的叶子结点存储的是主键的值
    • 基于非主键索引查询会产生回表
  • 降低回表次数
    • 使用联合索引，实现覆盖索引
  • 最左匹配原则
    • 因为索引是有序的，当匹配到范围查询等条件时，后续索引会失效
锁
  • 全局锁，表锁，行锁
